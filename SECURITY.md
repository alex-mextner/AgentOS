# Security Policy

## Reporting

Do not disclose an unpatched vulnerability in a public issue. Use GitHub's private vulnerability reporting or the repository Security tab when available. For non-sensitive security architecture discussion, open a public documentation issue.

## Scope

The project is in an early research and specification stage. No release should be treated as production-safe until its release notes explicitly state the supported hardware, threat model, update channel, and assurance level.

## Evidence expectations

Security fixes should include a minimal reproducer when safe, affected components, impact analysis, regression tests, and any required update or key-rotation procedure. Claims about secure boot, isolation, cryptography, or verified updates require inspectable evidence rather than design intent alone.
